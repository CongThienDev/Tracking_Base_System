Tracking Base System

Trang này giúp bạn hiểu hệ thống build để làm gì

Đây không phải website bán hàng hay dashboard BI. Đây là tầng hạ tầng tracking: nhận event từ app khác, lưu event chuẩn làm nguồn sự thật, rồi đẩy sang Meta/Google/TikTok theo cơ chế retry an toàn.

Mục tiêu chính

Một hành động thực tế của người dùng chỉ tạo ra 1 event_id, được lưu chuẩn 1 lần, và được fan-out xuống nhiều nền tảng mà không bị đếm trùng.

Dedup trước side effect Store-before-send Retry + trạng thái delivery Cutover an toàn

Hệ thống này KHÔNG làm gì

  • Không thay thế frontend app/product website.
  • Không phải dashboard báo cáo trực tiếp cho người dùng cuối.
  • Không phải nơi nhập tay dữ liệu marketing.

Luồng hoạt động end-to-end

1) Client AppBrowser / Backend gửi event
→
2) Tracking APIPOST /track validate + enrich
→
3) PostgresLưu canonical event + dedup bằng event_id
→
4) QueueBullMQ + Redis tạo job gửi destination
→
5) Router WorkerMap payload, retry, ghi trạng thái
→
6) Meta/Google/TikTokNhận conversion ổn định

Phase 7 hiện còn gì?

  • Đã xong: kickoff, baseline docs, UAT matrix.
  • Còn thiếu: chạy canary window và thu monitor evidence.
  • Còn thiếu: parity report thật từ dữ liệu canary vs baseline và sign-off.
  • Còn thiếu: rollback rehearsal có bằng chứng vận hành.
  • Cuối cùng: ghi quyết định go/no-go và chuyển Phase 7 sang done.
